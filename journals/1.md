## 2. Kenapa Bukan PHP?

Ini keputusan paling fundamental.

### PHP (cara lama — Jyavani CMS)

```
PHP (file .php) → PHP-FPM (interpreter) → CPU
```

Setiap request:
1. Nginx terima request
2. Nginx kirim ke PHP-FPM via unix socket
3. PHP-FPM baca file .php dari disk
4. PHP compile ke opcode (cache di OPcache)
5. Jalanin opcode
6. Balikin hasil ke Nginx

**Kelemahan:**
- **Interpreter wajib ada** — server harus install PHP-FPM
- **State antar request dipisah** — setiap request, PHP-FPM bisa jalan di proses berbeda
- **File .php harus ada di disk** — update = replace file
- **Scaling** — butuh PHP-FPM process manager (pm), urusan child process

### Go (cara baru — LEAH)

```
Go (kode .go) → go build (compiler) → binary (file) → CPU
```

Setelah build:
- Binary langsung jalan sebagai proses
- Di dalam binary udah termasuk: web server, routing, logic, semua dependency
- Gak perlu PHP, gak perlu interpreter, gak perlu web server tambahan

**Keuntungan:**
- **Zero runtime dependency** — cukup 1 file binary, jalan di OS mana pun
- **Startup cepat** — binary langsung jalan, gak perlu nunggu interpreter
- **Single process** — semua request ditangani sendiri (mirip Node.js event loop)
- **Goroutine** — ribuan request concurrent tanpa ribet manage process

### Filosofi yang Kupilih

> "Kontrol penuh, dependency minimal, satu binary untuk semuanya."

Aku suka bikin dari scratch (CMS native). Go kasih aku kontrol yang sama
seperti PHP native — tapi tanpa perlu interpreter di server.

## 3. Kenapa PostgreSQL, Bukan MariaDB/MySQL?

Aku selama ini pakai MariaDB untuk Jyavani CMS. Untuk LEAH, aku pilih PostgreSQL.

### Perbandingan

| Aspek | MariaDB | PostgreSQL |
|-------|---------|------------|
| **SQL compliance** | Baik | **Sangat baik** (lebih strict) |
| **JSON support** | Ada, tapi terbatas | **JSONB** — bisa index, query dalam JSON |
| **Array type** | ❌ | ✅ array native |
| **Full-text search** | Ada | **Lebih kuat** (tsvector) |
| **GIS/geolocation** | Plugin | **PostGIS** — standard industri |
| **Concurrency** | Baik | **Unggul** (MVVM, vacuum) |
| **Ekstensi** | Plugin | **Extension system** kuat (pg_stat_statements, uuid-ossp, dll) |
| **Cocok buat** | Web app sederhana | **Data kompleks, analitik, JSON** |

Kenapa ini penting untuk LEAH:
- **Asset inventory** — butuh JSON untuk metadata dinamis (tiap tipe aset punya field beda)
- **Full-text search** — cari ticket berdasarkan deskripsi
- **Analitik** — laporan jumlah ticket per bulan, per kategori, dll

> **Catatan:** Ini bukan berarti MariaDB jelek. Untuk CMS seperti Jyavani,
> MariaDB sudah lebih dari cukup. Tapi LEAH butuh fitur yang lebih advanced.

## 4. Arsitektur Handler → Service → Repository

Ini pola arsitektur yang kupakai. Tujuan: **pisahin tanggung jawab**.

```
HTTP Request
    ↓
Handler (menerima request, parse JSON, kirim response)
    ↓
Service (logic bisnis, validasi, aturan)
    ↓
Repository (SQL query ke database)
    ↓
PostgreSQL
```

### Kenapa dipisah?

| Layer | Tugas | Contoh |
|-------|-------|--------|
| **Handler** | Urusan HTTP | Baca body JSON, set status code 200/400/500 |
| **Service** | Aturan bisnis | "Kalau status kosong, isi 'open'" |
| **Repository** | Urusan database | SELECT, INSERT, UPDATE, DELETE |

**Manfaat:**
- Kalau mau ganti database (PostgreSQL → MySQL), cukup ganti repository
- Kalau mau ganti format response (JSON → XML), cukup ganti handler
- Service gak peduli database apa - dia cuma panggil function

Ini pola yang sama dengan yang biasa aku lakukan di CMS native,
cuma lebih terstruktur dan dipisah file.

## 5. Kenapa React Bukan Template PHP?

Di Jyavani, aku pakai PHP template (file .php di dashboard/theme/).
Di LEAH, aku pakai React. Kenapa?

### Cara Lama: Server-Side Rendering

```
User → Nginx → PHP-FPM → render HTML dari template → kirim ke browser
```

- HTML jadi di **server**
- Setiap klik link → request baru ke server
- Server render ulang seluruh halaman

### Cara Baru: Client-Side Rendering (React)

```
User → Browser → download React app → fetch data via API → render di browser
                                     ↓
                               Go API (backend) → JSON
```

- HTML jadi di **browser**
- Navigasi antar halaman **tanpa reload**
- Backend cuma kirim data (JSON), bukan HTML

### Kenapa ini penting untuk LEAH?

| Fitur | Server-Side (PHP) | Client-Side (React) |
|-------|-------------------|---------------------|
| Ganti halaman | Reload browser | Instant (tanpa reload) |
| Update data | Reload atau AJAX manual | **Auto-refresh** dengan React Query |
| Responsive | Butuh framework CSS manual | **Tailwind** — utility-first, cepat |
| User experience | Biasa | **Lebih responsif, seperti app** |

Tapi ini ada konsekuensi:
- **Butuh build step** — React harus di-compile jadi JS statis (`npm run build`)
- **SEO lebih kompleks** — tapi untuk internal tool (helpdesk), ini gak masalah
- **Dua bahasa** — Go (backend) + TypeScript (frontend)

### Filosofinya

> "Backend urus data, frontend urus tampilan. Mereka komunikasi lewat JSON API."

LEAH bukan website publik — ini internal tool. Jadi React dengan SPA (Single Page
Application) adalah pilihan tepat. Pengalaman user lebih mulus.

## 6. Cara LEAH Jalan di Production

Ini yang paling penting dipahami.

### Yang PERLU diinstall di server:

```
1. PostgreSQL    — database (wajib)
2. Nginx         — reverse proxy + SSL (wajib untuk production)
```

### Yang CUMA untuk build (bisa dihapus setelahnya):

```
1. Go           — compile backend jadi binary
2. Node.js      — build frontend React jadi file statis
```

### Cara update LEAH:

```bash
git pull                      # ambil kode terbaru
cd backend && go build -o leah ./cmd/api   # compile ulang binary
cd frontend && npm run build               # build ulang frontend
sudo systemctl restart leah                # restart service
```

Ada 3 langkah. Dulu PHP native tinggal `git pull` selesai.
Tapi kompensasinya: binary 16MB yang bisa jalan di mana pun tanpa install apa pun.

### Cara Kerja Binary Go

```
./leah (binary 16MB)
    │
    ├── Web server built-in (listen di port 8080)
    ├── Routing (Chi)
    ├── Logic bisnis (Service)
    ├── Database driver (pgx — PostgreSQL)
    └── Semua dependency (embedded dalam binary)
```

Binary ini **tidak butuh Go** untuk jalan. Dia sudah dalam bahasa mesin (machine code).
Sama seperti file `.exe` di Windows — tinggal klik, jalan.

## 8. Glossary — Istilah yang Perlu Diingat

| Istilah | Arti |
|---------|------|
| **Binary** | File executable hasil compile. Satu file = seluruh aplikasi |
| **Compile** | Proses ubah kode (.go) jadi instruksi mesin (binary) |
| **Handler** | Fungsi yang nerima HTTP request dan kirim response |
| **Service** | Logic bisnis — aturan aplikasi |
| **Repository** | Layer yang ngomong sama database (SQL) |
| **Router** | Penentu "request ini mau dikemanain" |
| **SPA** | Single Page Application — navigasi tanpa reload |
| **API** | Antarmuka untuk komunikasi antar aplikasi (biasanya JSON) |
| **Reverse proxy** | Nginx di depan, meneruskan request ke binary Go |
| **CORS** | Aturan keamanan browser untuk request lintas domain |
